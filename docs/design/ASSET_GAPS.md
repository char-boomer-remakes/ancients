# Asset Gaps: final visual audit

This audit is closed. The pass checked every screen-visible unit, prop, creature,
and marquee item against the GLBs we ship, then closed the real gaps:

1. entities still drawing a procedural rig when an authored model already exists,
2. shipped GLBs sitting unused, and
3. creep or hero mappings that read as the wrong creature family.

The standing rules still apply: original, generated, CC0, or CC-BY assets only;
never Valve or Blizzard files; every asset is optional at runtime; the game must
still boot and read cleanly with `public/assets/` empty. See `ASSETS.md` for the
ledger and `VFX_ASSETS.md` for the larger art-direction plan.

## Final State

| Surface | Current state |
|---|---|
| Heroes | 122/122 visually enhanced: 80 KayKit per-hero GLBs, 31 creature-base GLBs, 11 generated holdout replacement GLBs |
| Recruit NPCs | Use the same authored hero model they become after recruitment via `Unit.renderHeroId` |
| Creeps | Resolve through `creepCreatureUrl`; wrong-family reads from the audit are remapped |
| Creature assets | 23 creep GLBs are shipped and referenced by creeps, creature-base heroes, or ambient life |
| Towns | All 9 town GLBs are used, with procedural fallbacks and extra primitive dressing |
| Ambient life | `alpaca`, `fox`, and `frog` wander near towns as non-sim decorative critters |
| Items | Items remain procedural by design; 4 marquee artifacts now have generated signature held-weapon GLBs |
| Biomes | No open gap: all 6 biomes have terrain PBR, music, weather, and color grading |

`public/assets/manifest.json` now records 23 creep models, 9 town models, 80 hero
models, 22 holdout models, and 84 weapon models. The weapon count is 80 generated
hero weapons plus 4 generated item signatures.

## Shipped Work

### P0: Recruit NPCs

Recruitable heroes used to appear as flat procedural town standees, then snap to a
finished GLB once recruited. That broke identity at the exact moment the player
chose a hero.

Shipped fix:

- `Unit.renderHeroId` carries a render-only hero identity for NPCs.
- `spawnRecruitNpcs` sets `renderHeroId` from the region's `heroSpawns`.
- `scene.createView` treats `kind: 'hero'` and `kind: 'npc'` as hero-like when a
  render hero id exists.
- Per-hero GLBs, creature bases, holdout replacements, and holdout signature
  fallbacks all resolve through that render id.

Result: recruit NPCs use the same authored body they will use on the team, while
sim and AI logic stay untouched.

### P1: Creature GLB Coverage

The creature pass had two jobs: put every shipped creature GLB to work and fix the
worst wrong-species reads.

Shipped creep remaps:

| Creep | Previous read | Final read |
|---|---|---|
| `hellbear` | generic giant/brute | `bear` |
| `harpy-stormcrafter`, `harpy-scout` | grounded raptor | `flier` |
| `wildwing`, `wildwing-ripper`, `enraged-wildkin` | grounded raptor | `bear` |
| `ogre-bruiser` | same `orc` model as other brutes | `orcenemy` |
| `thunderhide`, `ancient-thunderhide` | bull-like beast | kept on `bull` |

Shipped hero-base remaps:

| Hero | Previous read | Final read |
|---|---|---|
| `ursa` | `giant` | `bear` |
| `tidehunter` | `giant` | `crab`, backed by `crabenemy.glb` |
| `treant-protector` | `giant` | `treant` |

Generated creature families:

- `flier.glb` covers harpy-style airborne creeps.
- `bear.glb` covers `ursa`, hellbears, and owlbear-style creeps.
- `treant.glb` covers `treant-protector`.

All three are original animated GLBs generated by
`scripts/assets/generate_creature_families.mjs`. They carry the same runtime
contract as the rest of the model layer: load when present, recolor per consumer,
mount over the procedural rig, and fall back silently if missing.

Every on-disk creature GLB is now consumed somewhere:

- Combat mappings use `bear`, `bull`, `demon`, `dragonevolved`, `flier`, `ghost`,
  `giant`, `glubevolved`, `goblin`, `golelingevolved`, `orc`, `orcenemy`, `stag`,
  `tribal`, `velociraptor`, `wolf`, and `yeti`.
- Creature-base heroes use `crabenemy`, `fox`, `spider`, and `treant`, plus many of
  the combat families above.
- Ambient life uses `alpaca`, `fox`, and `frog`.

The remaining creature notes are optional polish, not gaps: a walrus-specific
`tusk`, a better `snapfire` mount, a squirrel-specific `hoodwink`, a vehicle read
for `gyrocopter`, and a serpent/naga family if `medusa` or `naga-siren` ever need
a different generated replacement.

### P2: Towns And Ambient Life

Towns had more authored art than the scene used. The town pass wired the remaining
props and added small signs of life.

Shipped changes:

- `buildTownDressing` places the previously unused `well`, `cart`, `barrel`, and
  `market_stand_1` GLBs.
- Each authored prop has a primitive fallback so no-asset boot still looks
  intentional.
- Procedural lamp posts, a quest board, crates, a banner, and villager standees
  make the service hub readable in the world.
- `buildAmbientCritters` and `updateAmbientCritters` spawn non-sim `alpaca`, `fox`,
  and `frog` models around town with simple wandering, animation playback, ground
  seating, shared-asset disposal guards, and reduced-motion freezing.

Result: town assets are no longer stranded in the asset folder, and the overworld
has light ambient motion without adding sim entities.

### P3: Signature Item Weapons

Most item visuals should stay procedural. Components, consumables, and small stat
items are clearer as UI or VFX than as a cluttered pile of tiny held models.

The useful GLB upgrade was the marquee artifact tier. Shipped generated weapons:

- `daedalus.glb`
- `radiance.glb`
- `battlefury.glb`
- `divine-rapier.glb`

`itemWeaponGlbUrl` maps those item ids to
`public/assets/weapons/items/<id>.glb`. `applySignatureWeapon` and
`attachSignatureItemWeapon` mount the generated weapon over the hero's default hand
weapon when equipped, re-resolve on item changes, and restore the default weapon
when removed. Each item keeps its procedural `appearance.weapon` fallback.

## Verification Hooks

The closure is covered by targeted code paths and tests:

- `src/engine/assets.ts` owns `HERO_COHORTS`, `CREATURE_BY_ID`,
  `CREATURE_BY_BUILD`, `CREATURE_BASE_FILE`, and `itemWeaponGlbUrl`.
- `src/systems/game.ts` sets `renderHeroId` for recruit NPCs.
- `src/engine/scene.ts` mounts hero-like NPCs, shared creature bases, ambient
  critters, and signature item weapons.
- `src/engine/terrain.ts` builds the town dressing layer.
- `src/test/model-cache.test.ts` checks the generated `flier`, `bear`, and
  `treant` files, their creep and hero-base mappings, the 4 item weapon files, and
  item weapon mount/restore behavior.

Run the normal gates for any future asset batch:

```sh
npm run assets:check
npm run typecheck
npm test
npm run build
```

## Remaining Polish

These are taste upgrades, not blockers:

- Bespoke creature bases for `tusk`, `snapfire`, `hoodwink`, and `gyrocopter`.
- A serpent/naga family if the generated holdout replacements stop being enough.
- More generated signature item GLBs after the current four artifacts.
- A broader visual QA pass that screenshots recruit NPCs, town dressing, ambient
  critters, and artifact weapon swaps across quality tiers.
